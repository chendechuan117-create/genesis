# LESSON 候选：外部 provider 异常时的最小离线分流

**适用范围**
- 任何依赖外部 provider / 第三方 API / 远程模型服务的验证、诊断、交付或写入场景。
- 包括但不限于 429、5xx、网络失败、连接错误、读写超时、上游响应不稳定导致当前轮次无法安全完成的情况。

**触发条件**
- 本地可见日志、控制台或可复现请求显示外部 provider 出现 429、5xx、网络失败或超时。
- 或虽未精确定位错误码，但已有足够证据表明当前轮次因外部 provider 不稳定而无法继续安全验证或交付。
- 先核对真实日志与源码/文档中的异常处理机制，确认近期证据到底是什么异常；不要把未出现的错误码（如未证实的 502）写成既成事实。

**禁止动作**
- 禁止在证据未核实时，把单次 provider 故障扩写为系统性结论。
- 禁止继续宣称“已完成验证/已完成交付”，如果关键结果实际上仍依赖失稳 provider。
- 禁止在 provider 失稳期间继续做高风险写入、不可回滚修改或扩大影响面的操作。
- 禁止为追求闭环而硬凑结果、伪造成功、误引不存在的错误码或节点变化。

**默认降级产物**
- 进入最小离线分流，且优先只产出以下四类之一：
  1. 一句状态结论 + 明确重试建议；
  2. 只读诊断摘要；
  3. 一条可由人工执行的最小命令；
  4. 待 provider 恢复后再回写的草稿。
- 默认目标不是“继续完成原任务”，而是“在不误报、不高风险写入的前提下给出最小可交付物”。

**退出/恢复条件**
- provider 恢复稳定，且关键请求重新验证通过。**限定注：这里的“恢复稳定”不能仅由一次 `200 OK`、一次 failover success、一次切回首选 provider，或“流程继续”来判定；至少还需要连续观测证据，且不能把替代路径上的请求成功直接上升为原路径稳定、任务完成或服务等价。**
- 关键结果已被本地证据、重试结果或替代验证手段确认，不再依赖失稳状态下的猜测。
- 满足以上条件后，才可退出离线分流并恢复正常执行或写入。

**升级条件**
- 同类异常连续多次失败。
- 异常跨时段复现，表明并非偶发抖动。
- 异常已影响关键写入、关键交付、对外结论或多 provider 路径。
- 无法区分是 provider 配额/上游故障/网络问题/本地配置问题，需要更高层人工介入或正式事件处理。

**本地证据备注**
- 工作区本地日志可见：runtime/genesis.log 中存在 aixj.vip 连续 `HTTP/1.1 429 Too Many Requests` 记录。
- 另见 runtime/bot_restart.log：近期与 aixj.vip 相关的可见请求多为 `HTTP/1.1 200 OK`，并出现过 `httpx connection error (attempt 1/5)`。**限定注：这些 `200 OK` 样本本身最多只能证明对应请求成功返回；若缺少任务验收证据、原 provider 连续恢复证据或关键属性等价证据，不能单独推出“问题已解决”“原任务已完成”或“服务已等价恢复”。**
- 因此，本地更稳妥的可复用经验是“外部 provider 异常时先校正证据口径，再做最小离线分流”，而不是误称某单一错误码已再次被证实。
