# 治理基线草案

> 本文档为仓库治理加固第一轮初稿，依据 `ASSET_API_PROJECT_EVALUATION_QUESTIONS` 生成，目标是先建立最小可行治理框架，后续再逐步细化。

## 1. 文档依据
本治理基线主要参考以下评估维度：

1. **项目可用性与稳定性评估**
   - 项目活跃度
   - Issue 响应速度
   - 文档完整性和更新频率
   - 版本发布稳定性

2. **安全合规性问题**
   - API 密钥存储和加密方式
   - 访问控制和权限管理机制
   - 审计日志和监控功能
   - 数据隐私保护措施

3. **法律风险评估**
   - 服务条款合规性检查
   - 反爬虫策略应对
   - 使用限制和配额政策
   - 知识产权和版权问题

4. **国内平台特殊要求**
   - 实名认证流程和要求
   - 企业认证与个人认证差异
   - 数据出境合规性

5. **监控与告警机制**
   - 配额使用率监控
   - 异常访问检测
   - 多维度告警策略
   - 自动扩容和降级方案

## 2. 基本治理原则

### 2.1 文档与可维护性
- 仓库应维护基本治理文档，包括：`CONTRIBUTING.md`、`SECURITY.md`、`CODEOWNERS`、PR 模板
- 重要变更应同步更新 README、文档或运行说明
- 核心功能、配置方式和已知限制应可被新维护者快速理解

### 2.2 安全基线
- 严禁将敏感凭证提交到代码仓库
- 凭证应通过环境变量或受控配置注入
- 核心目录和敏感文件应有明确评审责任人
- 高风险变更应经过人工 Review

### 2.3 合规基线
- 接入第三方 API 前，应确认服务条款允许当前使用方式
- 对存在实名认证、企业认证、数据出境要求的平台，应先完成合规判断
- 不得默认实现规避平台限制、绕过风控或违反配额政策的逻辑

### 2.4 监控与可观测性
- 应逐步建立配额使用监控、异常访问检测和错误告警
- 关键任务应保留必要日志，支持问题回溯
- 日志不得输出敏感信息

### 2.5 变更管理
- 所有代码变更应通过 Pull Request 合并
- Pull Request 需说明变更目的、风险和测试情况
- 安全、权限、外部 API 接入类变更应显式说明影响范围

## 3. 当前最小落地要求
当前阶段，仓库至少满足以下要求：
- 已建立贡献指南
- 已建立安全策略文档
- 已建立 PR 模板
- 已建立基础 CODEOWNERS
- 新增变更遵循最小权限、最少暴露、最基本可审计原则

## 4. 后续建议迭代项
后续可继续补充：
- 行为准则（Code of Conduct）
- Issue 模板
- 版本发布流程
- 依赖安全扫描与 CI 检查
- 密钥扫描与提交钩子
- 支持版本与漏洞响应 SLA
- 第三方 API 合规清单

## 5. 说明
本文件是治理基线草案，不代表最终治理体系。其作用是先为仓库建立“有文档、可执行、可迭代”的基础框架。

## 6. 主库写入前质量闸门（SCAVENGED/低置信候选）

### 6.1 触发条件
当候选知识节点同时满足以下条件时，触发本闸门：
- `trust_tier=SCAVENGED`
- `confidence_score<=0.4` 或明确标记为低置信
- `validation_status=unverified`
- 正文过短，或明显为残片，或存在失真/乱码，或无法稳定复述其含义

### 6.2 阻断结论
- 满足上述触发条件的候选节点，一律不得直接写入可检索主库。

### 6.3 允许动作
触发本闸门后，只允许执行以下三类动作之一：
- 放弃写入
- 暂存为待验证线索
- 先补充最小可验证证据，完成复核后再升级

### 6.4 写入前最小检查清单
在任何放行判断前，必须逐项检查并记录：
- [ ] `trust_tier` 已核对
- [ ] `confidence_score` 已核对
- [ ] `validation_status` 已核对
- [ ] 正文可以被稳定复述，且不是残片、乱码或明显失真内容

### 6.4A 治理动作唯一分流判定入口（判定顺序）
对任一已发现事项，治理动作分流必须先经过本小节，且按以下顺序只选择一个单一最小落点，不得一轮并补多轨：
1. 若事项暴露出现行规则缺口、定义缺失或治理要求本身需要新增/修订，则唯一去向为治理基线修订；在规则未补齐前，不以 `KIL`、非 PR 结论性说明或 `LESSON` 代替规则修订。
2. 若事项仅属于命中 `6.5` 的记录触发情形，则唯一去向为 `KIL` 登记，并遵守 `6.5`、`6.7` 的先登记后宣告完成要求；`KIL` 只承担留痕与回查，不代替规则修订、非 PR 结论性说明或 `LESSON` 沉淀。
3. 若事项已经形成未通过 Pull Request 模板承载的结论、裁决、状态判断、规则例外说明或处理决定，则唯一去向为 `6.6` 所定义的非 PR 结论性说明对象，并按 `6.6`、`6.6.1` 满足最小字段要求；如同时命中 `6.5`，仍仅在该对象内回链对应 `KIL`，不得把 `KIL` 当作该说明的替代物。
4. 若事项已经被验证为可复用的经验、方法或行为约束，且不构成规则缺口修订、也不是某次具体结论性裁决或记录留痕，则唯一去向为 `LESSON` 沉淀；`LESSON` 只承载经验证可复用经验，不代替治理基线、`KIL` 或 `6.6` 对象。
上述四类对象彼此不可互相替代；判定时应优先选择能够完成本轮治理目的的单一最小落点，只有当后续出现新的独立触发事实时，才进入下一轨补充。

#### 6.4A.1 每轮行动前显式分流与落点声明
- 任何发现问题、提出修补、或准备调用 Op 的回合，均必须先按 `6.4A` 完成一次显式唯一分流，并在该回合输出中写明“本轮唯一落点”。
- 未完成上述分流与落点声明前，不得继续进入修订、记录、结论、或 `LESSON` 落地，也不得以并行补录方式同时推进多个落点。
- 若现有证据不足以支持唯一分流，必须立即停止当前推进，明确标记为“需要补证”或“需要升级判断”，不得在未定落点状态下继续多轨并补。

#### 6.4A.2 官方固定输出块最小母样（本轮唯一落点唯一标准承载位）
- 本小节所列输出块，是“本轮唯一落点”的唯一官方承载位与标准输出块，用于承载 `6.4A.1` 要求的回合级声明。
- 该输出块仅用于声明本回合的唯一分流结果，属于回合级声明；它不是 `KIL`，不是 Pull Request 模板，也不是 `6.6` 所定义的非 PR 结论性说明对象，三者不得互相替代。
- 行动性回合输出应在固定位置显式给出该输出块，优先写在开头；如未放在开头，也应保证人工检查时无需滚动查找即可直接看到。
- 官方最小母样如下，可直接抄写使用：

```md
[本轮唯一落点声明]
本轮唯一落点：治理基线修订
```

- 极简验收口径：人工检查时应能一眼看到“本轮唯一落点：____”字段；若缺失，则视为未完成 `6.4A.1` 声明。

### 6.5 KIL 统一触发与顺序约束母版条款
凡在任何承接面中出现以下任一情形，均必须触发 KIL：对知识对象作出 `approved` / `rejected` / `returned` 裁决；对待验证线索作出升级 / 退回 / 拒绝结论；留下任何可复用判断、状态裁决、规则例外说明的记录或说明。统一顺序要求为：必须先在固定登记面 `docs/knowledge_intake_log.md` 登记或补回对应记录锚点 `KIL-YYYYMMDD-序号`，并在相关记录、说明、提交或审计材料中引用 `docs/knowledge_intake_log.md#kil-YYYYMMDD-序号`；未完成登记并引用前，不得将上述动作宣告为已完成。

### 6.6 非 PR 结论性说明统一母版条款
- “非 PR 结论性说明”是指未通过 Pull Request 模板承载、但已经对某一对象形成可复用结论、裁决、状态判断、规则例外说明或处理决定的说明性内容；无论其出现于何种页面、记录、同步纪要、审计材料或其他承接面，只要可被后续引用、执行或复核，即属于本对象。
- 适用范围包括但不限于：知识对象或待验证线索的结论说明、复核结论、例外批准或拒绝说明、状态变更说明、针对规则适用结果形成的落地裁决说明。
- 明确排除范围：纯讨论、问题征询、未定稿草稿、临时同步口头纪要、仅用于过程沟通且未形成可复用裁决的备注、尚未形成明确结论或责任归属的工作中记录，不属于本条款所称非 PR 结论性说明。
- 任何非 PR 结论性说明均应使用统一最小字段母版，至少包含：结论/裁决、适用范围、依据或回链、状态、日期、责任人；字段暂缺时，不得留空或省略，必须显式标注“暂缺”，并说明暂缺原因。
- 本母版是页面无关的对象定义，只约束结论性说明本身必须具备的最小信息，不预设固定出口文件名，也不因承接页面不同而改变字段要求。
- 与 KIL 的关系边界如下：凡属于本条款定义的非 PR 结论性说明，且命中 6.5 所列任一触发情形的，必须能够回链到对应 `docs/knowledge_intake_log.md#kil-YYYYMMDD-序号` 锚点；在完成登记并完成该引用前，不得将该结论、裁决或状态变更宣告为已完成。
- 可目检的合规判定语句：若一份非 PR 说明已经形成可复用结论或裁决，则人工检查时应能直接看到“结论/裁决、适用范围、依据或回链、状态、日期、责任人”六项字段；如命中 6.5，还应能直接看到对应 KIL 锚点引用。缺任一项，或仅有结论而无字段、无回链、无暂缺原因者，均视为不合规。

#### 6.6.1 官方最小合规母样：非 PR 结论性说明示例
以下示例块为 `6.6` 的官方最小合规母样，也是人工判定非 PR 结论性说明是否合规的首个直接参照；承接页面可不同，但字段名与最小结构不得弱化。

```md
[非 PR 结论性说明] 节点复核裁决
结论/裁决：returned（暂不放行入主库，转回待验证线索）
适用范围：知识对象 `candidate-node-20250301-01` 的本次非 PR 复核结论说明
依据或回链：来源复核记录 2025-03-01 第 2 轮；规则依据见 `docs/governance_baseline.md#66-非-pr-结论性说明统一母版条款`；KIL 回链见 `docs/knowledge_intake_log.md#kil-20250301-01`
状态：已登记 KIL，待补充最小可验证证据后再复核
日期：2025-03-01
责任人：治理值班人 @example-owner
KIL 锚点（命中 6.5 时必须显式出现）：`docs/knowledge_intake_log.md#kil-20250301-01`
```

人工核对提示：应可直接逐项看到“结论/裁决、适用范围、依据或回链、状态、日期、责任人”六项字段；如该说明命中 `6.5`，还应显式看到 `KIL` 锚点字段或等价回链。

### 6.7 记录要求
- 命中本节闸门且需要作出 `approved`、`rejected` 或 `returned` 结论时，必须先在固定登记面 `docs/knowledge_intake_log.md` 逐条追加留痕，再执行写入、退回或拒绝动作。
- 固定回查方式：每次登记使用唯一记录锚点 `KIL-YYYYMMDD-序号`，后续说明、复核、提交或审计统一引用 `docs/knowledge_intake_log.md#kil-YYYYMMDD-序号`。
- 若最终放行，必须明确记录：为什么这次不属于上述阻断场景。
- 记录中至少应包含：日期、对象标识、来源凭证、验证/复核状态、结论、理由、执行者/责任人、关联规则锚点，以及 `trust_tier`、`confidence_score`、`validation_status`、正文质量判断、放行理由等必要上下文。

### 6.8 待验证线索闭环流转规则
- 触发本闸门后被暂存的内容，只能作为“待验证线索”存在；其定位是待核实工作项，不属于可检索主库知识，不得按正式知识节点参与检索、引用、复用或二次摘要。
- 待验证线索无论最终升级、退回还是拒绝，都必须能回查到对应登记记录；建议在升级说明或复核结论中直接引用 `docs/knowledge_intake_log.md#kil-YYYYMMDD-序号`。
- 待验证线索升级为主库知识前，至少补齐以下最小验证证据：一项可回查的原始来源凭证，以及一项能够证明正文含义未失真的复核记录；两项证据缺一不可。
- 只有同时满足以下条件，才允许触发升级：最小验证证据已补齐，关键信息字段完整，可稳定复述且不存在残片、乱码、明显失真或关键结论无法判定的情况。
- 待验证线索自暂存起，如在约定复核时限内仍未补齐证据，或复核后确认来源不可靠、内容失真、关键字段缺失、结论无法判定，则必须退场；退场方式只能是继续隔离待查或放弃写入，不得回流入主库。
- 线索最终放行进入主库时，必须同步记录放行理由，至少包括：采用了哪些验证证据、为何认定内容已可稳定复述、为何不再属于低置信阻断对象、由谁或按何次复核作出放行判断。
- 对缺字段、证据不足、无法判定、复核意见冲突但未消解的情形，默认策略一律为不得入主库。
